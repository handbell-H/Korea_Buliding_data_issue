# 우리나라 건축물 데이터의 매우 아쉬운 점

같은 "건물"을 다루는 데이터가 여러 개 있다.

문제는 데이터가 없다는 게 아니다. 데이터는 있다. 도로명주소 건물도 있고, GIS 건물정보도 있고, 건축물대장 속성정보도 따로 있다.

행안부, 국토지리정보원, 국토부 등 여러 기관이 각자의 목적에 맞춰 건축물 관련 데이터를 만든다. 도로명주소, 공간도형, 건축물대장, 건축물 속성처럼 담당 영역이 다른 것은 이해한다. 그런데 사용자 입장에서는 결국 같은 "건물"을 비교하고 싶은데, 자료마다 키와 형상이 달라서 매번 다시 해부해야 한다.

도형은 여기, 속성은 저기, 관리는 또 다른 곳. 이렇게 나뉘어 있더라도, 최소한 서로 이어서 비교할 수 있는 공통 실마리는 더 분명했으면 좋겠다.

세움터 원천자료 접근 방식이 예전과 달라진 건지, 내가 못 찾은 건지는 아직 모르겠다. 원천자료를 더 직접적으로 받을 수 있으면 해결될 것 같은데, 또 막상 받아도 다른 식별자 문제가 남아 있을 수 있다.

이 글은 그 답답함을 정리한 기록이다.

## 하고 싶었던 것

하고 싶었던 일은 단순했다.

```text
건물 간 비교를 정확히 해보고 싶다.
```

예를 들면 이런 질문이다.

```text
도로명주소 건물의 이 건물과
GIS건물정보의 이 건물과
건축물대장의 이 건물이
정말 같은 건물인가?
```

주용도와 연면적을 붙이는 문제가 아니다. GIS 건물정보에는 그런 속성이 이미 붙어 있는 경우도 있다. 핵심은 그 속성이 붙은 "이 건물"이 다른 데이터셋의 "그 건물"과 정말 같은지 확인하는 것이다.

## 어디서 막혔나

### 1. BD_MGT_SN은 만들 수 있을 것처럼 보인다

처음부터 `BD_MGT_SN`의 구조를 몰랐던 것은 아니다. 내가 사용한 도로명주소 건물 계열 SHP에서 확인한 `BD_MGT_SN`은 25자리였다.

예시:

```text
1153010200111310008022753
```

이 값은 다음처럼 볼 수 있다.

```text
1153010200111310008  PNU 계열 19자리
022753               건물관리번호로 보이는 6자리
```

즉:

```text
PNU 19자리 + 건물관리번호 6자리 = BD_MGT_SN 25자리
```

여기까지는 괜찮다. 문제는 건축물대장에도 비슷하게 "건물 하나를 가리키는 것처럼 보이는" 값이 있다는 점이다. 건축HUB 건축물대장 쪽에는 `관리_건축물대장_PK`가 있다.

예시:

```text
1002121184
1002114237
10021100185481
```

그러면 자연스럽게 이런 생각을 하게 된다.

```text
시군구코드
+ 법정동코드
+ 산/대지 구분
+ 본번
+ 부번
+ 관리_건축물대장_PK
= BD_MGT_SN을 만들 수 있지 않을까?
```

혹은:

```text
PNU 19자리
+ 관리_건축물대장_PK의 일부
= BD_MGT_SN 25자리?
```

하지만 내가 확인한 자료에서는 아니었다.

### 2. 관리_건축물대장_PK는 건물관리번호가 아니었다

같은 필지로 매칭된 건축물대장 행을 비교해보면, `관리_건축물대장_PK`는 `BD_MGT_SN`의 마지막 6자리와 맞지 않았다.

```text
BD_MGT_SN:
1153010200111310008022753

BD_MGT_SN 앞 19자리:
1153010200111310008

BD_MGT_SN 뒤 6자리:
022753

같은 필지로 매칭된 건축물대장 행의 관리_건축물대장_PK:
1018111539

관리_건축물대장_PK 뒤 6자리:
111539
```

`022753`과 `111539`는 다르다.

처음에는 `관리_건축물대장_PK`를 이용하면 `BD_MGT_SN`을 만들 수 있을 것 같았다. 하지만 실제로는 `BD_MGT_SN`의 건물관리번호와 다른 식별자처럼 작동했다. 이게 가장 큰 혼선이었다.

### 3. 비슷한 이름의 식별자가 너무 많다

자료를 보다 보면 다음 용어들이 계속 나온다.

```text
건물관리번호
건축물관리번호
건물식별번호
건축물대장관리번호
관리_건축물대장_PK
GIS건물통합식별번호
상위건물식별번호
고유번호(PNU)
```

이름만 보면 서로 이어질 것 같다. 하지만 내가 확인한 자료에서는 같은 키가 아니었다.

| 이름 | 내가 확인한 역할 |
|---|---|
| `BD_MGT_SN` | 도로명주소 건물 쪽 25자리 식별자 |
| `PNU` / `고유번호` | 필지 단위 19자리 식별자 |
| `관리_건축물대장_PK` | 건축물대장 쪽 내부 식별자로 보이는 값 |
| `건물식별번호` | GIS 건물자료 안에서 쓰이는 별도 식별자 |
| `GIS건물통합식별번호` | GIS 건물통합정보 계열의 별도 식별자 |

전부 건물과 관련 있어 보이지만, 비슷한 말이 같은 키라는 뜻은 아니었다.

### 4. PNU는 힌트지만 건물키는 아니다

`BD_MGT_SN`의 앞 19자리는 PNU/고유번호 계열과 비교할 수 있다.

```text
시군구코드 5자리
+ 법정동코드 5자리
+ 대지구분코드 1자리
+ 본번 4자리
+ 부번 4자리
= 19자리
```

하지만 필지는 건물이 아니다. 한 필지에 건물이 여러 개 있을 수 있고, 건축물대장 행도 여러 개일 수 있다. 그래서 필지키만으로 붙이면 "이 필지 위의 건물들"은 찾을 수 있지만, "이 건물 하나가 정확히 저 건물 하나다"라고 말하기 어렵다.

게다가 대지구분코드도 내가 사용한 자료 사이에서는 다르게 보였다.

```text
건축물대장 쪽:
0 = 대지로 보이는 일반 지번
1 = 산

BD_MGT_SN 앞 19자리/PNU 계열:
1 = 일반
2 = 산
```

이 차이를 보정하지 않고 그대로 붙이면, 내가 비교한 파일에서는 매칭이 0건이 됐다.

### 5. 공간으로 비교해도 끝나지 않는다

키가 안 맞으면 공간으로 비교할 수도 있다.

```text
건물 폴리곤끼리 겹쳐 보고
중심점이 들어가는지 보고
면적 중첩률을 계산하고
가장 많이 겹치는 객체를 같은 건물로 본다
```

이론상 가능하다. 그런데 실제로는 이것도 깔끔하지 않다.

내가 확인한 파일에서는 같은 건물처럼 보이는 객체도 출처에 따라 형상이 달랐다. 어떤 자료는 건물 외곽을 더 단순하게 갖고 있고, 어떤 자료는 더 잘게 나뉘어 있고, 어떤 자료는 부속건물이나 연결부를 다르게 처리한다. 경계가 살짝 어긋나기도 하고, 하나의 건물이 여러 폴리곤으로 쪼개져 있거나 반대로 여러 건물이 하나처럼 보이는 경우도 있다.

즉 공간상에서 비교할 수는 있지만, 그것도 "정확한 건물 간 비교"라기보다는 확률 높은 후보를 찾는 과정에 가깝다.

## 확인한 범위

이 글은 모든 건축물 데이터셋의 공식 명세를 해설하려는 글이 아니다. 내가 실제로 내려받아 비교한 서울 지역 자료와, 그 과정에서 확인한 식별자 문제를 정리한 작업 기록에 가깝다.

내가 가진 파일 기준으로는 `관리_건축물대장_PK`를 이용해 `BD_MGT_SN`을 만들 수 없었고, `관리_건축물대장_PK` 뒤 6자리도 `BD_MGT_SN` 뒤 6자리와 맞지 않았다. `BD_MGT_SN` 앞 19자리와 PNU/고유번호 계열은 일부 비교 가능했지만, 그것만으로 건물 단위 정확 매칭이 되는 것은 아니었다.

모든 지역과 모든 시점의 자료에서 같은 현상이 반복되는지는 확인하지 않았다. 별도의 비공개 또는 별도 신청 연계자료에 직접 매핑키가 있는지도 확인하지 않았다.

## 결론

내가 겪은 우리나라 건축물 데이터의 문제는 데이터가 없다는 게 아니다. 데이터는 있다.

문제는 같은 건물을 가리키는 것처럼 보이는 식별자가 자료마다 다르고, 그 식별자들이 실제로는 서로 바로 이어지지 않는다는 것이다.

특히 `BD_MGT_SN`의 뒤 6자리 건물관리번호와 `관리_건축물대장_PK`가 이어질 것 같다는 착시가 컸다. 하지만 내가 확인한 자료에서는 그 둘은 직접 매칭되지 않았다.

그래서 건물 단위 비교를 하려면 데이터를 받는 것보다 먼저 식별자 체계를 해부해야 한다.

이게 제일 아쉽다.
