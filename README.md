# 우리나라 건축물 데이터의 빡치는 점

같은 “건물”을 다루는 데이터가 여러 개 있다.

문제는 데이터가 없다는 게 아니다. 데이터는 있다. 도로명주소 건물도 있고, GIS 건물정보도 있고, 속성정보만 따로 있기도 하다.

세움터 원천자료 접근 방식이 예전과 달라진 건지, 내가 못 찾은 건지는 아직 모르겠다. 원천자료를 더 직접적으로 받을 수 있으면 해결될 것 같은데, 또 막상 받아도 다른 식별자 문제가 남아 있을 수 있다.

어쨌든 우리나라 건축물 데이터는 서로 이어서 비교하려고 하면 갑자기 막힌다.

행안부, 국토지리정보원, 국토부 등 여러 기관이 각자의 목적에 맞춰 건축물 관련 데이터를 만든다. 도로명주소, 공간도형, 건축물대장, 에너지·녹색건축 관련 속성처럼 담당 영역이 다른 것은 이해한다. 그런데 사용자 입장에서는 결국 같은 “건물”을 비교하고 싶은데, 자료마다 키와 형상이 달라서 매번 다시 해부해야 한다.

도형은 여기, 속성은 저기, 관리는 또 다른 곳. 이렇게 나뉘어 있더라도, 최소한 서로 이어서 비교할 수 있는 공통 실마리는 더 분명했으면 좋겠다.

각 데이터셋이 쓰는 키가 다르고, 비슷한 이름의 식별자가 여러 개 나오고, 어떤 키가 어떤 키와 이어지는지 한눈에 알 수 없다. 용어는 전부 건물 비슷한 말을 하는데, 실제로는 서로 다른 체계다.

이 문서는 그 답답함을 정리한 기록이다.

## 먼저 적어두는 검증 범위

이 글은 모든 건축물 데이터셋의 공식 명세를 완전히 해설하려는 글이 아니다.

내가 실제로 내려받아 비교한 서울 지역 자료와, 그 과정에서 확인한 식별자 문제를 정리한 것이다. 따라서 아래 내용은 “내가 사용한 파일과 시점에서는 이렇게 보였다”는 작업 기록에 가깝다.

특히 다음 내용은 실제 파일을 기준으로 확인했다.

```text
BD_MGT_SN 전체 25자리와 관리_건축물대장_PK는 직접 매칭되지 않았다.
BD_MGT_SN 뒤 6자리와 관리_건축물대장_PK 뒤 6자리도 매칭되지 않았다.
BD_MGT_SN 앞 19자리는 PNU/고유번호 계열과 일부 매칭됐다.
공간 비교는 가능하지만, 출처별 건물 형상이 달라 예외 처리가 필요했다.
```

즉 이 글의 핵심은 “절대 불가능하다”가 아니라, “공식 자료를 받아도 데이터셋 간 건물 단위 비교키가 명확하게 드러나지 않아 사용자가 직접 검증해야 했다”는 것이다.

반대로, 아직 확인하지 않은 것도 있다.

```text
모든 지역과 모든 시점의 자료에서 같은 현상이 반복되는지는 확인하지 않았다.
별도의 비공개 또는 별도 신청 연계자료에 직접 매핑키가 있는지는 확인하지 않았다.
각 식별자의 법적/행정적 정의 전체를 해설하려는 글은 아니다.
```

그래서 이 글은 결론을 단정하기보다는, 공개 자료를 실제로 이어 붙이려는 사용자가 어디서 막히는지를 보여주는 쪽에 가깝다.

## 내가 하고 싶었던 것

하고 싶었던 일은 단순했다.

```text
건물 간 비교를 정확히 해보고 싶다.
```

예를 들면 이런 질문이다.

```text
도로명주소 건물의 이 건물과
GIS건물정보의 이 건물과
건축물대장의 이 건물이
정말 같은 건물인가?
```

이걸 확인하고 싶었다.

주용도와 연면적을 붙이는 문제가 아니다. GIS 건물정보에는 그런 속성이 이미 붙어 있는 경우도 있다. 문제는 그 속성이 붙은 “이 건물”이 다른 데이터셋의 “그 건물”과 정말 같은지 확인하는 것이다.

그런데 실제로 해보면 단순하지 않다.

## 문제 1. 건물을 가리키는 키가 자료마다 다르다

내가 사용한 도로명주소 건물 계열 SHP에서는 `BD_MGT_SN`을 확인했다.

예시:

```text
1153010200111310008022753
```

내가 확인한 값은 25자리 문자열이었고, 예시 기준으로는 다음처럼 나눠볼 수 있었다.

```text
11530  시군구코드
10200  법정동코드
1      대지구분코드
1131   번
0008   지
022753 건물번호로 보이는 마지막 6자리
```

반면 건축HUB 건축물대장 쪽에는 `관리_건축물대장_PK`가 있다.

예시:

```text
1002121184
1002114237
10021100185481
```

길이도 제각각이다.

GIS건물통합정보나 GIS 건물일반집합정보 쪽에는 또 다른 식별자들이 나온다.

```text
GIS건물통합식별번호
고유번호(PNU)
건물식별번호
상위건물식별번호
```

전부 그럴듯하다.

전부 건물과 관련 있어 보인다.

그런데 이 중 무엇이 `BD_MGT_SN`과 같은 건물을 가리키는지 바로 알 수 없다.

## 문제 2. 용어가 비슷해서 더 헷갈린다

다음 용어들은 다 비슷하게 생겼다.

```text
건물관리번호
건축물관리번호
건물식별번호
건축물대장관리번호
관리_건축물대장_PK
GIS건물통합식별번호
상위건물식별번호
```

이름만 보면 서로 이어질 것 같다.

하지만 실제 값은 서로 다르다.

내가 비교한 샘플에서는 `BD_MGT_SN`의 마지막 6자리와 `관리_건축물대장_PK`의 마지막 6자리가 맞지 않았다.

```text
BD_MGT_SN:
1153010200111310008022753

BD_MGT_SN 뒤 6자리:
022753

같은 필지로 매칭된 건축물대장 행의 관리_건축물대장_PK:
1018111539

PK 뒤 6자리:
111539
```

비슷한 이름이라고 같은 체계가 아니다.

이게 꽤 치명적이다.

## 문제 3. 필지키는 만들 수 있지만 건물키는 아니다

내가 확인한 `BD_MGT_SN` 값에서는 앞 19자리를 필지 단위 키처럼 쓸 수 있었다.

```text
시군구코드 5자리
+ 법정동코드 5자리
+ 대지구분코드 1자리
+ 번 4자리
+ 지 4자리
= 19자리
```

이 19자리 키는 PNU 계열 고유번호와 어느 정도 이어진다.

하지만 필지는 건물이 아니다.

한 필지에 건물이 여러 개 있을 수 있다. 반대로 건축물대장 행도 여러 개일 수 있다. 그래서 필지키만으로 붙이면 “이 필지 위의 건물들”은 찾을 수 있지만, “이 건물 하나가 정확히 저 건물 하나다”라고 말하기 어렵다.

즉:

```text
필지 단위 비교는 가능
건물 단위 정확 비교는 어려움
```

## 문제 4. 코드 체계도 미묘하게 다르다

대지구분코드도 내가 사용한 자료 사이에서는 다르게 보였다.

건축물대장 쪽에서는:

```text
0 = 대지로 보이는 일반 지번
1 = 산
```

`BD_MGT_SN` 앞 19자리/PNU 계열에서는:

```text
1 = 일반
2 = 산
```

이런 차이는 작아 보이지만, 조인할 때는 치명적이다.

이 차이를 보정하지 않고 그대로 붙이면, 내가 비교한 파일에서는 매칭이 0건이 됐다.

## 문제 5. “통합정보”라고 해도 내가 원하는 비교키가 있는 건 아니다

GIS건물통합정보나 GIS 건물일반집합정보는 이름만 보면 모든 문제가 해결될 것 같다.

```text
GIS
건물
통합
정보
```

이름이 아주 든든하다.

하지만 실제로 필요한 것은 이거다.

```text
이 데이터셋의 건물과
저 데이터셋의 건물이
같은 건물인지 판단할 수 있는 키
```

그런데 막상 자료를 보면 각자 자기 식별자를 들고 있다.

```text
BD_MGT_SN
GIS건물통합식별번호
고유번호(PNU)
건물식별번호
관리_건축물대장_PK
```

하지만 내가 받은 파일 기준으로는 이 값들이 서로 어떤 관계인지 사용자가 다시 검증해야 했다.

적어도 내가 하려던 `도로명주소 건물 - GIS 건물정보 - 건축물대장` 간 비교에서는, “통합”이라는 말이 곧바로 쓸 수 있는 공통 비교키를 의미하지는 않았다.

## 문제 6. 공간상으로 비교할 수는 있지만, 형상도 출처마다 다르다

물론 키가 안 맞으면 공간으로 비교할 수도 있다.

```text
건물 폴리곤끼리 겹쳐 보고
중심점이 들어가는지 보고
면적 중첩률을 계산하고
가장 많이 겹치는 객체를 같은 건물로 본다
```

이론상 가능하다.

그런데 실제로는 이것도 깔끔하지 않다.

내가 확인한 파일에서는 같은 건물처럼 보이는 객체도 출처에 따라 형상이 달랐다. 어떤 자료는 건물 외곽을 더 단순하게 갖고 있고, 어떤 자료는 더 잘게 나뉘어 있고, 어떤 자료는 부속건물이나 연결부를 다르게 처리한다. 경계가 살짝 어긋나기도 하고, 하나의 건물이 여러 폴리곤으로 쪼개져 있거나 반대로 여러 건물이 하나처럼 보이는 경우도 있다.

즉 공간 비교도 정답 버튼은 아니다.

```text
키 매칭이 어렵다
-> 공간 매칭을 해본다
-> 형상 기준도 출처마다 다르다
-> 다시 임계값과 예외 처리가 필요하다
```

결국 공간상에서 비교할 수는 있지만, 그것도 “정확한 건물 간 비교”라기보다는 확률 높은 후보를 찾는 과정에 가깝다.

## 실제로 확인한 것

작업하면서, 내가 가진 파일 기준으로 확인한 내용은 이렇다.

```text
BD_MGT_SN 전체 25자리
-> 건축물대장 관리_건축물대장_PK와 직접 매칭 안 됨

BD_MGT_SN 뒤 6자리
-> 관리_건축물대장_PK 뒤 6자리와 매칭 안 됨

BD_MGT_SN 앞 19자리
-> PNU/고유번호 계열과는 일부 매칭 가능

PNU/고유번호
-> 필지 단위라 건물 단위 정확 매칭은 아님

공간 비교
-> 가능은 하지만 출처마다 건물 형상이 달라 임계값과 예외 처리가 필요함
```

즉, 가장 현실적인 연결은 이런 식이다.

```text
BD_MGT_SN에서 앞 19자리 필지키를 뽑는다
-> GIS 건물자료의 PNU 계열 키와 맞춘다
-> 그 자료 안의 다른 건물 식별자를 본다
-> 건축물대장 PK와 맞는지 다시 검증한다
-> 그제야 서로 같은 건물인지 비교한다
```

이게 한 번에 되는 일이 아니다.

자료를 받을 때마다 키의 의미와 매칭률을 확인해야 했다.

## 왜 이게 빡치냐

건물 데이터 분석에서 가장 기본적인 질문은 이거다.

```text
이 공간정보의 건물이
다른 데이터셋의 어떤 건물인가?
```

그런데 내가 확인한 자료 조합에서는 이 질문에 바로 답할 수 있는 키가 잘 보이지 않았다.

자료마다 건물을 다루고, 이름도 비슷하고, 설명도 그럴듯하다. 하지만 실제로는 식별자 체계가 서로 다르다.

결국 사용자는 다음 일을 해야 한다.

```text
1. 각 자료의 키를 뜯어본다.
2. 자리수를 센다.
3. 앞자리와 뒷자리를 잘라본다.
4. 코드 체계를 비교한다.
5. 필지 단위로 붙여본다.
6. 공간상으로 겹쳐본다.
7. 형상 차이 때문에 다시 의심한다.
8. 건물 단위로 맞는지 검토한다.
9. 매칭률을 계산한다.
```

이건 데이터 분석이라기보다 식별자 추리 게임에 가깝다.

## 결론

내가 겪은 우리나라 건축물 데이터의 문제는 데이터가 없다는 게 아니다.

데이터는 있다.

문제는 같은 건물을 가리키는 식별자가 자료마다 다르고, 그 식별자들 사이의 관계가 사용자가 바로 알 수 있을 만큼 명확하지 않다는 것이다.

도로명주소 건물, GIS건물통합정보, GIS 건물일반집합정보, 건축HUB 총괄표제부는 모두 건물을 다룬다.

하지만 이 네 자료를 이어서 비교하려고 하면, 내가 확인한 범위에서는 바로 쓸 수 있는 공통 키가 보이지 않았다.

비슷한 용어는 많다.
