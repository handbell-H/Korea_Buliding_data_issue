# 우리나라 건축물 데이터의 매우 아쉬운 점

같은 “건물”을 다루는 데이터가 여러 개 있다.

문제는 데이터가 없다는 게 아니다. 데이터는 있다. 도로명주소 건물도 있고, GIS 건물정보도 있고, 속성정보만 따로 있기도 하다.

그런데 서로 이어서 비교하려고 하면 갑자기 막힌다.

처음부터 `BD_MGT_SN`의 구조를 몰랐던 것은 아니다. `BD_MGT_SN`이 대략 다음 구조라는 것은 알고 있었다.

```text
PNU 19자리 + 건물관리번호 6자리 = BD_MGT_SN 25자리
```

문제는 그 다음이다.

건축물대장에는 `관리_건축물대장_PK`가 있고, GIS 건물자료에는 `건물식별번호`가 있고, 도로명주소 건물에는 `건물관리번호`가 있다. 이름만 보면 서로 이어질 것 같다.

그래서 처음에는 이렇게 생각하게 된다.

```text
시군구코드
+ 법정동코드
+ 산/대지 구분
+ 본번
+ 부번
+ 관리_건축물대장_PK
= BD_MGT_SN을 만들 수 있지 않을까?
```

혹은:

```text
PNU 19자리
+ 관리_건축물대장_PK의 일부
= BD_MGT_SN 25자리?
```

하지만 실제로는 아니었다.

`관리_건축물대장_PK`는 `BD_MGT_SN`의 마지막 6자리인 건물관리번호가 아니었다. `건물식별번호`도 이름은 비슷하지만, 내가 확인한 자료 조합에서는 `BD_MGT_SN`의 건물관리번호와 바로 대응되는 키로 쓸 수 없었다.

이 문서는 그 혼선을 정리한 기록이다.

## 먼저 적어두는 검증 범위

이 글은 모든 건축물 데이터셋의 공식 명세를 완전히 해설하려는 글이 아니다.

내가 실제로 내려받아 비교한 서울 지역 자료와, 그 과정에서 확인한 식별자 문제를 정리한 것이다. 따라서 아래 내용은 “내가 사용한 파일과 시점에서는 이렇게 보였다”는 작업 기록에 가깝다.

특히 다음 내용을 실제 파일 기준으로 확인했다.

```text
BD_MGT_SN = PNU 19자리 + 건물관리번호 6자리 구조로 볼 수 있다.
건축물대장의 관리_건축물대장_PK는 BD_MGT_SN의 건물관리번호가 아니었다.
관리_건축물대장_PK의 뒤 6자리도 BD_MGT_SN의 뒤 6자리와 맞지 않았다.
GIS 건물자료의 건물식별번호도 BD_MGT_SN과 직접 매칭되는 키로 확인되지 않았다.
결국 내가 가진 공개 자료 조합만으로는 BD_MGT_SN과 건축물대장 PK를 바로 이어주는 키를 찾지 못했다.
공간 비교는 가능하지만, 출처별 건물 형상이 달라 예외 처리가 필요했다.
```

즉 이 글의 핵심은 “BD_MGT_SN을 모른다”가 아니다.

핵심은:

```text
BD_MGT_SN의 건물관리번호
관리_건축물대장_PK
건물식별번호
GIS건물통합식별번호
```

이런 비슷한 이름의 식별자들이 서로 같은 것처럼 보이지만, 실제로는 같은 키가 아니었다는 점이다.

반대로, 아직 확인하지 않은 것도 있다.

```text
모든 지역과 모든 시점의 자료에서 같은 현상이 반복되는지는 확인하지 않았다.
별도의 비공개 또는 별도 신청 연계자료에 직접 매핑키가 있는지는 확인하지 않았다.
각 식별자의 법적/행정적 정의 전체를 해설하려는 글은 아니다.
```

그래서 이 글은 결론을 단정하기보다는, 공개 자료를 실제로 이어 붙이려는 사용자가 어디서 헷갈리고 막히는지를 보여주는 쪽에 가깝다.

## 내가 하고 싶었던 것

하고 싶었던 일은 단순했다.

```text
건물 간 비교를 정확히 해보고 싶다.
```

예를 들면 이런 질문이다.

```text
도로명주소 건물의 이 건물과
GIS건물정보의 이 건물과
건축물대장의 이 건물이
정말 같은 건물인가?
```

주용도와 연면적을 붙이는 문제가 아니다. GIS 건물정보에는 그런 속성이 이미 붙어 있는 경우도 있다. 문제는 그 속성이 붙은 “이 건물”이 다른 데이터셋의 “그 건물”과 정말 같은지 확인하는 것이다.

그런데 실제로 해보면 단순하지 않다.

## 문제 1. `BD_MGT_SN`은 만들 수 있을 것처럼 보인다

내가 사용한 도로명주소 건물 계열 SHP에서는 `BD_MGT_SN`을 확인했다.

예시:

```text
1153010200111310008022753
```

이 값은 다음처럼 볼 수 있다.

```text
1153010200111310008  PNU 계열 19자리
022753               건물관리번호로 보이는 6자리
```

즉:

```text
PNU 19자리 + 건물관리번호 6자리 = BD_MGT_SN 25자리
```

여기까지는 괜찮다.

문제는 건축물대장에도 비슷하게 “건물 하나를 가리키는 것처럼 보이는” 값이 있다는 점이다.

건축HUB 건축물대장 쪽에는 `관리_건축물대장_PK`가 있다.

예시:

```text
1002121184
1002114237
10021100185481
```

이 값을 보면 자연스럽게 이런 생각을 하게 된다.

```text
PNU 19자리 + 관리_건축물대장_PK 일부 = BD_MGT_SN?
```

하지만 실제로는 아니었다.

## 문제 2. `관리_건축물대장_PK`는 `건물관리번호`가 아니다

내가 비교한 샘플에서는 `관리_건축물대장_PK`가 `BD_MGT_SN`의 마지막 6자리와 맞지 않았다.

```text
BD_MGT_SN:
1153010200111310008022753

BD_MGT_SN 앞 19자리:
1153010200111310008

BD_MGT_SN 뒤 6자리:
022753

같은 필지로 매칭된 건축물대장 행의 관리_건축물대장_PK:
1018111539

관리_건축물대장_PK 뒤 6자리:
111539
```

`022753`과 `111539`는 다르다.

즉:

```text
관리_건축물대장_PK != BD_MGT_SN의 건물관리번호
관리_건축물대장_PK의 뒤 6자리 != BD_MGT_SN의 뒤 6자리
```

처음에는 `관리_건축물대장_PK`를 이용하면 `BD_MGT_SN`을 만들 수 있을 것 같았지만, 실제로는 `BD_MGT_SN`과 상관없는 별도 식별자처럼 작동했다.

이게 가장 큰 혼선이었다.

## 문제 3. `건물관리번호`, `건물식별번호`, `건축물대장_PK`가 모두 다르게 보인다

다음 용어들은 다 비슷하게 생겼다.

```text
건물관리번호
건축물관리번호
건물식별번호
건축물대장관리번호
관리_건축물대장_PK
GIS건물통합식별번호
상위건물식별번호
```

이름만 보면 서로 이어질 것 같다.

하지만 실제 값은 서로 다르다.

내가 확인한 자료에서는 대략 이런 식으로 갈라졌다.

```text
BD_MGT_SN
-> 도로명주소 건물 쪽 25자리 건물관리번호

PNU/고유번호
-> 필지 단위 19자리 식별자

관리_건축물대장_PK
-> 건축물대장 내부 식별자로 보이는 값

건물식별번호
-> GIS 건물자료 안에서 쓰이는 별도 식별자

GIS건물통합식별번호
-> GIS 건물통합정보 계열의 별도 식별자
```

전부 건물과 관련 있어 보이지만, 같은 키는 아니었다.

이름이 비슷해서 더 헷갈린다.

## 문제 4. 필지키는 만들 수 있지만 건물키는 아니다

`BD_MGT_SN`의 앞 19자리는 PNU/고유번호 계열과 비교할 수 있다.

```text
시군구코드 5자리
+ 법정동코드 5자리
+ 대지구분코드 1자리
+ 본번 4자리
+ 부번 4자리
= 19자리
```

하지만 필지는 건물이 아니다.

한 필지에 건물이 여러 개 있을 수 있다. 반대로 건축물대장 행도 여러 개일 수 있다. 그래서 필지키만으로 붙이면 “이 필지 위의 건물들”은 찾을 수 있지만, “이 건물 하나가 정확히 저 건물 하나다”라고 말하기 어렵다.

즉:

```text
필지 단위 비교는 가능
건물 단위 정확 비교는 어려움
```

필지키는 힌트이지, 건물 단위 정답 키가 아니다.

## 문제 5. 코드 체계도 미묘하게 다르다

대지구분코드도 내가 사용한 자료 사이에서는 다르게 보였다.

건축물대장 쪽에서는:

```text
0 = 대지로 보이는 일반 지번
1 = 산
```

`BD_MGT_SN` 앞 19자리/PNU 계열에서는:

```text
1 = 일반
2 = 산
```

이런 차이는 작아 보이지만, 조인할 때는 치명적이다.

이 차이를 보정하지 않고 그대로 붙이면, 내가 비교한 파일에서는 매칭이 0건이 됐다.

## 문제 6. “통합정보”라고 해도 내가 원하는 비교키가 있는 건 아니다

GIS건물통합정보나 GIS 건물일반집합정보는 이름만 보면 모든 문제가 해결될 것 같다.

```text
GIS
건물
통합
정보
```

이름이 아주 든든하다.

하지만 실제로 필요한 것은 이거다.

```text
이 데이터셋의 건물과
저 데이터셋의 건물이
같은 건물인지 판단할 수 있는 키
```

그런데 막상 자료를 보면 각자 자기 식별자를 들고 있다.

```text
BD_MGT_SN
GIS건물통합식별번호
고유번호(PNU)
건물식별번호
관리_건축물대장_PK
```

하지만 내가 받은 파일 기준으로는 이 값들이 서로 어떤 관계인지 사용자가 다시 검증해야 했다.

적어도 내가 하려던 `도로명주소 건물 - GIS 건물정보 - 건축물대장` 간 비교에서는, “통합”이라는 말이 곧바로 쓸 수 있는 공통 비교키를 의미하지는 않았다.

## 문제 7. 공간상으로 비교할 수는 있지만, 형상도 출처마다 다르다

물론 키가 안 맞으면 공간으로 비교할 수도 있다.

```text
건물 폴리곤끼리 겹쳐 보고
중심점이 들어가는지 보고
면적 중첩률을 계산하고
가장 많이 겹치는 객체를 같은 건물로 본다
```

이론상 가능하다.

그런데 실제로는 이것도 깔끔하지 않다.

내가 확인한 파일에서는 같은 건물처럼 보이는 객체도 출처에 따라 형상이 달랐다. 어떤 자료는 건물 외곽을 더 단순하게 갖고 있고, 어떤 자료는 더 잘게 나뉘어 있고, 어떤 자료는 부속건물이나 연결부를 다르게 처리한다. 경계가 살짝 어긋나기도 하고, 하나의 건물이 여러 폴리곤으로 쪼개져 있거나 반대로 여러 건물이 하나처럼 보이는 경우도 있다.

즉 공간 비교도 정답 버튼은 아니다.

```text
키 매칭이 어렵다
-> 공간 매칭을 해본다
-> 형상 기준도 출처마다 다르다
-> 다시 임계값과 예외 처리가 필요하다
```

결국 공간상에서 비교할 수는 있지만, 그것도 “정확한 건물 간 비교”라기보다는 확률 높은 후보를 찾는 과정에 가깝다.

## 실제로 확인한 것

작업하면서, 내가 가진 파일 기준으로 확인한 내용은 이렇다.

```text
BD_MGT_SN은 PNU 19자리 + 건물관리번호 6자리 구조로 볼 수 있었다.

시군구코드 + 법정동코드 + 산/대지 구분 + 본번 + 부번 + 관리_건축물대장_PK
-> 이 조합으로 BD_MGT_SN을 만들 수 있을 것처럼 보였지만 아니었다.

관리_건축물대장_PK
-> BD_MGT_SN의 건물관리번호와 직접 매칭되지 않았다.

관리_건축물대장_PK 뒤 6자리
-> BD_MGT_SN 뒤 6자리와 매칭되지 않았다.

건물식별번호 / GIS건물통합식별번호
-> 이름은 비슷하지만 BD_MGT_SN과 직접 연결되는 키로 확인되지 않았다.

BD_MGT_SN 앞 19자리
-> PNU/고유번호 계열과는 일부 매칭 가능했다.

PNU/고유번호
-> 필지 단위라 건물 단위 정확 매칭은 아니었다.

공간 비교
-> 가능은 하지만 출처마다 건물 형상이 달라 임계값과 예외 처리가 필요했다.
```

즉, 가장 큰 문제는 `BD_MGT_SN`의 앞 19자리를 모르는 것이 아니었다.

진짜 문제는 뒤 6자리 건물관리번호와 연결될 것처럼 보이는 `관리_건축물대장_PK`, `건물식별번호` 등이 실제로는 같은 키가 아니었다는 점이다.

## 왜 이게 빡치냐

건물 데이터 분석에서 가장 기본적인 질문은 이거다.

```text
이 공간정보의 건물이
다른 데이터셋의 어떤 건물인가?
```

그런데 내가 확인한 자료 조합에서는 이 질문에 바로 답할 수 있는 키가 잘 보이지 않았다.

자료마다 건물을 다루고, 이름도 비슷하고, 설명도 그럴듯하다. 하지만 실제로는 식별자 체계가 서로 다르다.

결국 사용자는 다음 일을 해야 한다.

```text
1. BD_MGT_SN을 분해한다.
2. PNU 19자리와 건물관리번호 6자리를 나눈다.
3. 관리_건축물대장_PK가 건물관리번호와 관련 있는지 의심한다.
4. 실제로 비교해보면 관련이 없다는 걸 확인한다.
5. 건물식별번호, GIS건물통합식별번호도 확인한다.
6. 그래도 직접 비교키가 보이지 않는다.
7. 필지 단위로 붙여본다.
8. 공간상으로 겹쳐본다.
9. 형상 차이 때문에 다시 의심한다.
10. 건물 단위로 맞는지 검토한다.
```

이건 데이터 분석이라기보다 식별자 추리 게임에 가깝다.

## 결론

내가 겪은 우리나라 건축물 데이터의 문제는 데이터가 없다는 게 아니다.

데이터는 있다.

문제는 같은 건물을 가리키는 것처럼 보이는 식별자가 자료마다 다르고, 그 식별자들이 실제로는 서로 바로 이어지지 않는다는 것이다.

도로명주소 건물, GIS건물통합정보, GIS 건물일반집합정보, 건축HUB 총괄표제부는 모두 건물을 다룬다.

하지만 이 네 자료를 이어서 비교하려고 하면, 내가 확인한 범위에서는 바로 쓸 수 있는 공통 키가 보이지 않았다.

비슷한 용어는 많다.

```text
건물관리번호
건물식별번호
건축물대장 PK
GIS건물통합식별번호
PNU
```

그런데 비슷한 말이 같은 키라는 뜻은 아니다.

특히 `BD_MGT_SN`의 뒤 6자리 건물관리번호와 `관리_건축물대장_PK`가 이어질 것 같다는 착시가 컸다. 하지만 내가 확인한 자료에서는 그 둘은 직접 매칭되지 않았다.

그래서 건물 단위 비교를 하려면 데이터를 받는 것보다 먼저 식별자 체계를 해부해야 한다.

이게 제일 빡친다.
